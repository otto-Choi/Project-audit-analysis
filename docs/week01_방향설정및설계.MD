# 외부감사 데이터 분석 시스템 — 설계 결정 과정

> 이 문서는 시스템 설계 과정에서 어떤 이유로 어떤 결정을 내렸는지를 기록한 것입니다.  
> 코드나 수식보다는 **왜 이런 구조를 선택했는가**에 초점을 맞춥니다.

---

## 1. 프로젝트 배경 및 목표

### 왜 이 프로젝트를 시작했는가

외부감사 실무에서 감사인은 회사로부터 다양한 형태의 자료를 수령하고, 이를 분석하여 감사 절차를 수행한다. 그러나 현실적으로 두 가지 큰 어려움이 존재한다.

첫째, **감사대상기업마다 데이터 형태가 완전히 다르다.** 같은 "계정별 원장"이라도 컬럼명, 날짜 형식, 금액 단위, 파일 구조가 전부 제각각이다. 매번 수작업으로 이를 정리하는 데 상당한 시간이 소요된다.

둘째, **회계법인의 보안 규범이 매우 엄격하다.** 외부 서비스나 클라우드를 활용할 수 없기 때문에, 직접 개발한 로컬 도구가 아니면 실무에 바로 투입할 수 없다.

### 목표

- 파이썬 학습을 겸하여 회사 제시 자료를 가공하는 능력을 기른다.
- 다양한 데이터 형태에 유연하게 대응할 수 있는 구조를 설계한다.
- 회계법인 보안 환경에서 즉시 활용 가능한 로컬 도구를 구축한다.
- 감사 절차에서 기계적으로 반복하는 작업을 자동화하여 감사인이 판단에 집중할 수 있게 한다.

### 핵심 원칙 (설계 전체를 관통하는 기준)

이 시스템의 모든 설계 결정은 아래 다섯 가지 원칙에서 출발한다.

1. **원본 데이터는 절대 건드리지 않는다** — 감사 증거로서의 원본 보존
2. **회사별 설정은 config로, 분석 로직은 재사용** — 클라이언트가 바뀌어도 로직은 동일
3. **모든 처리는 로컬에서 완결** — 보안 규범 준수
4. **조서 작성은 감사인 영역** — 오류 가능성 때문에 조서 자동 출력은 하지 않음
5. **검증 가능한 처리** — 모든 가공 과정은 로그와 대사 수식으로 추적 가능

## 2. 데이터셋 선택 과정

### 검토한 데이터셋들

시스템 개발 연습용으로 여러 캐글 데이터셋을 검토했다.

**SAP BigQuery Dataset (ska1 등)**  
SAP FI 모듈의 실제 테이블 구조(BKPF, BSEG, SKA1 등)를 담고 있어 JET(분개 테스트) 수준의 분석 연습에 적합하다. 전기자 ID, 문서유형, T-code 등 감사에 핵심적인 필드가 포함되어 있다. 다만 계정담당자 수준의 연습보다는 JET 전담팀 수준의 분석에 더 맞는 데이터다.

**irfansharif/generalledger**  
단순 총계정원장 수준의 데이터로, 이미 정리된 집계형 자료다. 전기자나 문서유형 같은 세부 필드는 없지만, 실제 계정담당자가 회사로부터 받는 자료의 수준과 유사하다. 분석적 절차나 추세 분석 연습에 적합하다.

**aidsmlprojects SAP FI anomaly detection**  
SAP BKPF/BSEG 기반의 합성 데이터로, 이상치 레이블까지 포함되어 있다. 그러나 이 데이터셋은 ML 모델 성능 평가를 위해 설계된 것이어서, 감사인이 스스로 "이게 왜 이상한가"를 판단하는 훈련에는 오히려 적합하지 않다. 정답이 레이블로 주어져 있기 때문이다.

### 결론

캐글 데이터는 실제 클라이언트 자료를 받기 전에 다양한 형태의 데이터를 다루는 연습용 샘플로만 활용한다. 목적에 따라 적합한 데이터셋이 다르다.

- 계정담당자 수준 연습 → irfansharif/generalledger
- JET 수준 연습 → SAP BigQuery Dataset

### 계정담당자가 실제로 받는 자료 수준에 대한 이해

외부감사에서 JET 팀은 BKPF/BSEG 수준의 상세 데이터를 요청할 수 있지만, 인건비·판관비 등을 담당하는 계정담당자가 실제로 수령하는 자료는 **원장 수준**에 그치는 경우가 대부분이다. 이 현실을 반영하여 시스템을 설계한다.

## 3. 파이프라인 구조 결정

### 3단 분리 구조를 채택한 이유

가장 핵심적인 설계 결정이다. 전체 파이프라인을 세 단계로 분리한다.

```
[1단: 변환] → [2단: 정제] → [3단: 분석]
```

이 구조를 채택한 이유는 **클라이언트마다 다른 것과 항상 같아야 하는 것을 명확히 분리**하기 위해서다.

- **변환 단계**: 회사가 준 엑셀을 CSV로 바꾸는 것. 클라이언트마다 다르다.
- **정제 단계**: CSV를 표준 스키마로 만드는 것. 설정(config)만 다르고 로직은 동일하다.
- **분석 단계**: 정제된 데이터를 분석하는 것. 항상 동일하게 재사용된다.

1단계만 클라이언트별로 수정하면 2·3단계는 그대로 쓸 수 있다.

### 엑셀을 직접 분석하지 않고 CSV를 거치는 이유

처음에는 엑셀 파일을 pandas로 직접 읽는 방안을 검토했다. 그러나 실제 엑셀 파일에는 수많은 함정이 있다.

- 회사 로고나 제목 때문에 헤더가 5행부터 시작하는 경우
- 날짜나 계정코드 셀이 병합되어 아래 행이 빈값으로 들어오는 경우
- 숨겨진 행/열이 데이터에 포함되는 경우
- 엑셀이 날짜를 시리얼 숫자로 저장하는 경우
- 금액 셀이 수식으로 되어 있는 경우

엑셀에서 CSV로 수동 변환하면 이 중 상당수가 해결된다(엑셀이 렌더링한 값 그대로 저장). 나머지는 변환 모듈에서 처리한다.

단, CSV 변환 시 **반드시 UTF-8 (BOM 포함) 형식**으로 저장해야 한글 깨짐이 없다.

### 변환 config를 별도로 두는 이유

같은 회사에서 수정본을 받거나 다른 계정 보조부를 받을 때, 회사의 엑셀 양식은 동일하다. 따라서 변환 설정을 한 번만 작성해두면 이후에는 재사용할 수 있다. 변환 config와 정제 config를 분리함으로써 이 재사용성을 확보한다.

## 4. 필드 스키마 결정

### 기본 필드 선정

감사 업무에서 최소한 필요한 필드를 먼저 정했다. 날짜, 계정코드, 차변, 대변, 원장번호, 적요가 출발점이었다.

여기에 감사인 관점에서 추가로 필요한 필드를 검토했다.

- **전기일과 문서일의 구분**: 거래 발생일(문서일)과 장부 반영일(전기일)이 다를 수 있다. 마감 전후 전기 분석에서 이 구분이 중요하다. 단, 없으면 문서일로 대체한다.
- **계정명**: 코드만으로는 분석 결과를 이해하기 어렵다. 단, 원장의 계정명은 행마다 다르게 기재되는 경우가 많아 마스터 기준으로 덮어쓴다.
- **거래처 정보**: 특수관계자 거래 식별에 필수
- **부서/코스트센터**: 비용의 부서별 배분 합리성 검토
- **전기자 ID**: 권한 외 전기 탐지. 단, 원장 수준 자료에서는 없는 경우가 많다.

### Tier 구조로 분류한 이유

모든 필드를 필수로 요구하면 자료 수령 자체가 어려워진다. 반대로 아무것도 요구하지 않으면 분석이 제한된다. 따라서 세 등급으로 분류했다.

- **Tier 1 (필수)**: 없으면 분석 자체가 불가능한 필드
- **Tier 2 (권장)**: 있으면 분석 범위가 크게 확장되는 필드
- **Tier 3 (선택)**: JET 수준이지만 받을 수 있으면 활용

## 5. 금액 형태 처리 결정

### 4가지 유형을 분류한 이유

실제 회사 자료를 보면 금액이 표현되는 방식이 매우 다양하다.

- **A형 (차대변 분리)**: 차변 컬럼과 대변 컬럼이 별도로 존재. 가장 일반적.
- **B형 (부호형)**: 하나의 금액 컬럼에 양수/음수로 차대변을 표현.
- **C형 (플래그형)**: 금액 컬럼과 별도의 구분 플래그 컬럼이 존재. "D/C", "차/대", "1/2" 등 회사마다 다름.
- **D형 (잔액 누계형)**: 잔액이 누적으로 기재되어 전기 잔액과의 차이로 역산해야 함.

유형을 config에서 선택하면, 이후 분석 모듈은 항상 A형(차대변 분리)으로 정규화된 데이터만 받는다.

**C형에서 플래그 컬럼명을 명시해야 하는 이유**  
처음에는 플래그 값(D/C 등)만 지정했으나, 어느 컬럼이 플래그인지도 지정해야 한다는 것을 뒤늦게 파악했다. 원본에 컬럼이 여러 개 있을 때 자동으로 특정할 수 없기 때문이다.

### 부호 방향 설정이 필요한 이유

양수가 항상 차변을 의미하지 않는다. 매출 계정처럼 성격에 따라 반대인 경우도 있다. 따라서 부호 방향을 config에서 명시적으로 지정한다.

### 괄호형 음수를 별도로 처리하는 이유

엑셀에서 회계 서식으로 저장된 음수는 `(1,000)` 형태로 CSV에 기재된다. 이는 문자열로 인식되어 숫자 변환 시 오류가 발생한다. 흔한 케이스이므로 별도 처리 옵션을 둔다.

## 6. 날짜 형식 처리 결정

### 다양한 날짜 형식을 모두 지원하는 이유

실무에서 마주치는 날짜 형식은 매우 다양하다.

- `2024-01-15` (하이픈형)
- `2024.01.15` (점형, 한국 공문서 스타일)
- `20240115` (8자리 숫자, ERP 직접 추출 시 흔함)
- `45306` (엑셀 시리얼 숫자, 날짜 셀이 숫자로 저장된 경우)
- `2024년 1월 15일` (한글형)

특히 **엑셀 시리얼 숫자**는 함정이다. 엑셀에서 날짜처럼 보이는 셀이 CSV로 저장될 때 숫자로 변환되는 경우가 있어, 별도 처리 로직이 필요하다.

**연/월/일 순서 설정이 필요한 이유**  
`01/15/2024`(미국식)와 `15/01/2024`(유럽식)는 숫자만 봐서는 구분이 불가능하다. 따라서 config에서 순서를 명시해야 한다.

## 7. 마스터 데이터 설계 결정

### 계정 마스터를 별도 파일로 관리하는 이유

원장 데이터에 계정명이 포함되어 있더라도 신뢰할 수 없는 경우가 많다. 같은 계정코드인데 "매출액", "매 출 액", "매출" 등 행마다 다르게 기재되는 일이 실제로 흔하다. 따라서 원장의 계정명은 무시하고, 별도로 관리하는 계정 마스터 기준으로 덮어씌운다.

계정 마스터에 없는 코드가 원장에 나타나면 즉시 경고를 발생시킨다. 이것 자체가 감사 포인트가 될 수 있다.

### 계정 계층구조(parent_code)를 설계한 이유

실무에서 재무제표상 "접대비"라는 항목 아래에 "접대비-일반(31101)", "접대비-기타(31102)"처럼 세분화된 계정이 존재한다. 증감분석이나 샘플링에서는 이 세부 계정들을 하나로 묶어서 봐야 하는 경우가 많다.

`parent_code` 컬럼으로 상위 계정을 연결해두면, 분석 목적에 따라 세부 계정 수준과 집계 수준을 자유롭게 선택할 수 있다.

### `is_posting` 컬럼이 필요한 이유

계층구조에서 집계용 계정(상위 계정)에는 실제 전기가 이루어지지 않는다. 이를 구분하지 않으면 집계가 중복 계산될 수 있다. `is_posting`이 `Y`인 계정만 필터링하면 실제 전기 데이터만 분석 대상으로 삼을 수 있다.

### 비용 구분(cost_type, cost_detail) 컬럼을 추가한 이유

손익계산서에서 동일한 계정(예: 급여)이 제조원가와 판관비로 나뉘어 배분되는 경우가 있다. 이를 구분하지 않으면 매출원가와 판관비를 각각 집계하는 것이 불가능하다. 계정 마스터에서 이 배분을 사전에 정의해두면 분석 단계에서 자동으로 구분된다.

### 거래처 마스터를 별도로 관리하는 이유

특수관계자 거래 식별이 주목적이다. 거래처명만으로는 특수관계자 여부를 판단할 수 없으므로, 별도 마스터에 관계 유형을 정의해두고 원장과 연결한다.

## 8. 소계행 처리 결정

### 삭제가 아닌 분리로 처리하는 이유

소계행을 단순 삭제하면 실제 데이터 행을 잘못 삭제한 경우를 감지하기 어렵다. 따라서 소계행으로 판단된 행을 별도 데이터프레임으로 분리하고, 소계행의 금액 합계와 데이터행의 합계가 일치하는지 검증한다. 불일치하면 실제 데이터 누락 가능성을 경고한다.

### 소계행 탐지에 복수 조건을 사용하는 이유

단일 조건으로는 탐지가 불완전하다. 소계행의 전형적인 패턴은 다음과 같다.

- 날짜나 계정코드 같은 필수 컬럼이 비어있는 경우
- 적요에 "합계", "소계", "계" 같은 키워드가 있는 경우
- 계정코드 자릿수가 세부 계정보다 짧은 경우(상위 계정이 소계로 사용)

이 세 조건의 OR 조합으로 탐지하면 대부분의 소계행을 잡을 수 있다.

## 9. 데이터 완전성 검증을 설계한 이유

받은 데이터가 올바른지 확인하지 않으면 이후 분석 전체가 의미없어진다. 따라서 정제 전에 아래를 먼저 확인한다.

- **기간 완전성**: 1월~12월 요청했는데 특정 월 데이터가 누락된 경우
- **차대변 균형**: 전표 단위로 차변 합계 = 대변 합계인지 확인
- **중복 행**: 동일 전표번호가 두 번 들어온 경우
- **시산표 대사**: 원장 합계가 회사 제출 시산표와 일치하는지

## 10. 참조데이터 엑셀 파일 구조 결정

### 왜 하나의 엑셀 파일로 통합했는가

조서에서 XLOOKUP/SUMIFS로 참조할 때 파일이 여러 개면 관리가 복잡해진다. 하나의 파일에 모든 참조 데이터를 담으면 조서에서 단일 파일만 참조하면 된다.

### 시트 구성을 결정한 과정

**BS/PL 원본 시트를 별도로 두는 이유**  
회사가 제출한 원본 자료를 그대로 보존하기 위해서다. 가공하지 않고 그대로 붙여넣는다.

**GL 원본과 GL_가공을 분리한 이유**  
처음에는 원본 GL에 파생변수를 우측에 추가하는 방안을 검토했다. 그러나 원본 컬럼 수가 회사마다 달라서 파생변수 위치가 항상 달라지고, 이렇게 되면 수식을 매번 수정해야 한다. 원본과 파생변수를 완전히 다른 시트로 분리하면 GL_가공 시트의 파생변수는 항상 A열부터 고정된다.

**GL_가공에 원본 컬럼 참조를 넣지 않기로 한 이유**  
처음에는 파생변수 다음에 원본 컬럼을 수식으로 참조하는 방안을 검토했다. 그러나 GL_가공 시트는 파생변수만 담당하는 것이 역할 분리 측면에서 더 명확하다. 원본이 필요하면 GL 원본 시트를 직접 보면 된다.

**_idx 컬럼을 원본 GL에 추가한 이유**  
원본 GL과 GL_가공을 연결할 키가 필요하다. 원장 데이터에는 계정코드가 있지만 동일 계정의 행이 여러 개이므로 유일 키로 쓸 수 없다. 전표번호도 라인 수준에서는 중복될 수 있다. 따라서 Python이 1부터 순번을 부여하는 `_idx`를 원본 GL에만 추가한다. 이 하나의 컬럼만 원본에 추가하는 것은 원본 보존 원칙에서 예외적으로 허용한다.

**TB(시산표) 시트를 별도로 두는 이유**  
BS/PL 원본은 회사마다 양식이 달라서 수식으로 직접 참조하면 수식이 매번 달라진다. TB에서 규격화된 형태로 한 번 정리해두면, 수식모음의 XLOOKUP/SUMIFS 수식이 회사 불문 항상 동일하게 유지된다. TB는 허브 역할을 한다.

**TB를 수식이 아닌 값으로 채우기로 한 이유**  
BS/PL 원본을 수식으로 연결하면 원본 양식이 바뀔 때마다 수식을 수정해야 한다. Python pandas가 BS/PL를 읽어서 값으로 직접 기입하면 수식 오류 위험이 없고, 버전이 바뀌면 Python 재실행만 하면 된다.

### 시트명에 날짜를 포함하는 이유

버전이 바뀔 때(수정본 수령 시) 새 파일을 만들면 시트명의 날짜만 달라진다. 조서에서 파일 참조 경로만 변경하면 수식 전체가 자동으로 새 버전 데이터를 반조한다.

## 11. 수식모음 설계 결정

### 수식모음 시트를 Python이 자동 생성하는 이유

수식을 매번 직접 작성하면 오탈자나 참조 오류가 발생할 수 있다. 시트명이 포함된 수식을 Python이 자동 생성하면 항상 정확한 시트명이 들어가고, 새 버전을 받았을 때도 재실행만 하면 된다.

### 검증 수식을 수식모음에 포함한 이유

TB와 GL_가공이 값으로 채워지기 때문에 Python 처리 과정에서 오류가 발생해도 육안으로 발견하기 어렵다. 검증 수식을 통해 원본과 가공 데이터가 일치하는지 엑셀에서 직접 확인할 수 있다.

검증 항목은 다음과 같다.
- GL 원본 vs GL_가공: 행 수, 차변 합계, 대변 합계, 순액, _idx 연속성
- BS 원본 vs TB: 자산/부채/자본 합계, BS 등식(자산=부채+자본)
- PL 원본 vs TB: 수익/비용 합계, 당기순이익
- GL_가공 vs TB: diff 합계(0이어야 함), GL 순액과 TB gl_amount 일치

BS/PL 원본 셀 참조 부분은 양식이 회사마다 달라 감사인이 수동으로 지정해야 한다. 나머지는 전부 자동이다.

### 분석 수식이 XLOOKUP과 SUMIFS로 나뉘는 이유

- **XLOOKUP**: TB에서 계정별 1:1 조회. 증감액, 증감률 등 계정당 하나의 값을 가져올 때.
- **SUMIFS**: GL_가공에서 조건부 집계. 같은 계정이 여러 행에 걸쳐 있거나, 매출원가/판관비처럼 배분이 필요할 때.

PL 분석에서 SUMIFS를 주로 쓰는 이유는, 동일한 급여 계정이 제조원가와 판관비로 나뉘어 배분되는 경우 XLOOKUP으로는 집계가 불가능하기 때문이다.

## 12. 파생변수를 GL_가공 A열부터 배치하는 이유

원본 GL의 컬럼 수가 회사마다 다르다. 만약 파생변수를 원본 컬럼 다음에 배치하면 파생변수의 열 위치가 회사마다 달라진다. 이렇게 되면 수식에서 열을 참조할 때 매번 수정이 필요하다.

파생변수를 항상 A열부터 고정 배치하면, 어떤 클라이언트의 데이터를 쓰더라도 수식이 동일하다. GL_가공 시트의 파생변수 이후에 원본 컬럼을 넣지 않기로 한 것도 같은 이유다.

## 13. 클라이언트 특화 컬럼 처리 결정

### 왜 필요한가

특정 사업부 자료만 분석 대상으로 삼거나, 회사 고유의 분류 컬럼(프로젝트코드, 품목코드 등)을 활용해야 하는 경우가 있다. 이를 config에서 명시적으로 지정하면 분석 로직을 수정하지 않고도 클라이언트별 특화 분석이 가능하다.

### 설계 방식

원본 컬럼명과 GL_가공에서 쓸 alias를 config에서 지정한다. 파생변수 다음에 추가되므로 열 위치가 뒤로 밀리지만, 표준 파생변수 위치는 영향받지 않는다.

필터 조건도 config에서 지정하면 특정 사업부 데이터만 정제 대상으로 삼을 수 있다.

## 14. 버전 관리 설계 결정

### 왜 버전 관리가 필요한가

외부감사에서 회사는 초기 제출 후 오류 수정이나 추가 조정사항을 반영하여 수정된 BS/PL 및 원장을 재제출하는 경우가 많다. 이때 감사인은 버전 간 차이가 무엇인지 파악해야 한다.

### 수령일 기준 폴더 구조를 택한 이유

버전 번호보다 수령일이 더 직관적이고 감사 증거로서도 의미 있다. 언제 어떤 자료를 받았는지가 감사 기록에 중요하다.

### 원본 파일을 항상 raw 폴더에 보존하는 이유

어떤 처리를 하더라도 원본은 변경되지 않아야 한다. 처리 오류가 발생했을 때 언제든지 원본으로 돌아갈 수 있어야 하며, 감사 증거로서 원본 보존이 필수다.

## 15. 조서 자동 출력을 하지 않기로 한 결정

### 왜 조서 자동 출력을 포기했는가

처음에는 openpyxl을 이용해 조서를 자동 생성하는 방안을 검토했다. 그러나 다음과 같은 이유로 포기했다.

- 조서는 전기 양식을 따르는 경우가 많아 형태와 절차가 클라이언트마다 매우 상이하다.
- 조서에는 감사인의 판단, 결론, 서명 등이 포함되어야 하는데 이는 자동화할 수 없다.
- 자동 생성된 조서에 오류가 있을 경우 감사 품질에 직접적인 위험이 된다.

따라서 Python은 분석과 탐색까지만 담당하고, 조서 작성은 감사인의 영역으로 명확히 분리했다. Python의 결과물은 조서에 붙여넣거나 XLOOKUP으로 참조할 수 있는 깔끔한 참조 데이터 수준으로만 제공한다.

## 16. 개발 환경 결정

### Jupyter Notebook을 개발 도구로 선택한 이유

파이썬 학습이 목적 중 하나이므로, 코드와 결과가 같은 화면에 보이는 Jupyter가 학습에 유리하다. 코드를 셀 단위로 실행하며 중간 결과를 확인할 수 있어 디버깅이 쉽다.

### Streamlit을 탐색 UI로 선택한 이유

"특정 원장 데이터를 타고 들어가며 살펴보는" 기능이 필요했다. 매출 원장에서 특정 전표를 클릭하면 해당 분개 전체를 볼 수 있고, 분개에서 매출채권 계정을 클릭하면 매출채권 원장 전체를 볼 수 있는 드릴다운 탐색이다. 이런 클릭 기반 탐색은 터미널이나 Jupyter만으로는 구현하기 어렵다.

Streamlit은 로컬에서 완전히 실행되며 인터넷이 필요 없어 보안 규범에 부합한다.

### Jupyter에서 Streamlit을 실행하지 않는 이유

Streamlit은 독립 실행 앱이기 때문에 Jupyter 커널 내에서 실행하면 다른 셀을 사용할 수 없게 된다. 코드 수정도 앱을 껐다 켜야 해서 비효율적이다. 따라서 역할을 명확히 분리한다.

- **Jupyter**: 코드 개발, 학습, 심층 분석
- **Streamlit**: 완성된 분석 결과 탐색 UI

Jupyter에서 개발하고 검증한 로직을 Streamlit 파일로 옮기는 흐름이 자연스럽다.

### 핵심 라이브러리를 최소화한 이유

회계법인 환경에서는 인터넷이 차단된 경우가 많아 패키지 설치가 제한될 수 있다. pandas, openpyxl, matplotlib, streamlit 정도의 표준적인 라이브러리로도 목적하는 기능 전부를 구현할 수 있으므로 복잡한 ML 라이브러리는 도입하지 않는다.
