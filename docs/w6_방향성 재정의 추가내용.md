# 1. 방향성 재정의 배경

## 초기 프로젝트 목표

본 프로젝트는 원래 외부감사 실무에서 발생하는 반복 작업을 줄이기 위한 목적에서 시작되었다.

초기 목표는 다음에 가까웠다.

```text
SAP 원장 데이터 병합
→ 감사 분석용 정제
→ 계정 탐색
→ Excel 조서 출력
````

즉:

* 감사 데이터 정제
* 원장 탐색
* 조서 자동화

등 감사 실무 중심의 자동화 프로젝트 성격이 강했다.

---

## 프로젝트 진행 중 발견한 문제

프로젝트를 진행하면서 단순 자동화보다 더 큰 문제가 존재한다는 점을 확인하게 되었다.

대표적으로:

* 클라이언트마다 완전히 다른 ERP 추출 구조
* 상이한 컬럼명·금액 형태·날짜 형식
* 서로 다른 리스크 판단 기준
* 구조화되지 않은 원장 데이터
* 탐색과 리스크 검토 workflow 부재

등이다.

즉 문제의 본질은:
“Excel 자동화”
보다

```text
비정형 ERP 데이터를 어떻게 표준화하고,
어떻게 리스크 기반으로 탐색 가능한 구조로 만들 것인가
```

에 더 가까웠다.

---

## 방향성 변경

따라서 프로젝트 방향을 다음과 같이 재정의한다.

### 기존 방향

```text
감사 자동화 도구
```

---

### 변경 후 방향

```text
ERP 기반 Risk Analytics Pipeline
```

---

## 현재 프로젝트 핵심 구조

```text
ERP 데이터
  ↓
표준화 및 정제
  ↓
Risk Detection
  ↓
Risk Scoring
  ↓
Drill-down Investigation
```

# 3. notebook ↔ src 역할 분리

## 현재 문제점

현재 프로젝트는 notebook 중심 구조의 흔적이 일부 남아 있다.

문제:

* notebook 내부에 비즈니스 로직 존재
* 재사용 가능한 함수와 실험 코드 혼재
* 유지보수성과 구조 명확성 저하 가능

---

## 역할 분리 기준

### notebooks/

역할:

* 실행
* 검증
* 실험
* 샘플 출력
* 시각화

즉:
“오케스트레이션 레이어”

---

### src/

역할:

* 실제 비즈니스 로직
* Rule 처리
* 데이터 변환
* reusable 함수

즉:
“비즈니스 로직 레이어”

---

## 목표 구조

```text
notebooks/
  w6_anomaly.ipynb

src/
  anomaly.py
```

---

## 구현 원칙

notebook 내부에는:

```python
from src.anomaly import apply_rules
```

수준의 호출만 남긴다.

실제 Rule 로직은 src로 이동한다.

# 4. Risk / Flag Naming 표준화

## 문제점

Week 6부터 `_flag_*` 컬럼이 급격히 증가한다.

Naming 규칙이 흔들리면:

* filtering
* aggregation
* Streamlit 연동
* export
* risk score 계산

전체 구조가 복잡해질 가능성이 높다.

---

## 표준 Naming 규칙

### Flag

```text
_flag_<rule_name>
```

예시:

```text
_flag_weekend_posting
_flag_month_end_posting
_flag_duplicate_journal
_flag_large_manual_entry
```

---

### Risk

```text
_risk_score
_risk_level
_risk_flags
```

---

### 기타 파생변수 Prefix

| Prefix   | 의미      |
| -------- | ------- |
| `_flag_` | 이상탐지    |
| `_risk_` | 위험도     |
| `_post_` | 날짜 파생   |
| `_is_`   | bool 상태 |

# 5. Risk Score 기준 단위 확정

## 핵심 설계 이슈

Rule은:

* line 기준
* document 기준

둘 다 가능하다.

하지만 혼재되면:

* score 계산
* aggregation
* Streamlit UI
* reviewer workflow

가 복잡해질 수 있다.

---

## 최종 방향

### Document 기준 Risk Aggregation 채택

```text
라인 단위 flag 생성
  ↓
doc_no 기준 집계
  ↓
전표(document) 기준 risk_score 계산
```

---

## 이유

감사 실무의 실제 검토 단위는:

* 개별 line보다
* “전표(document)”인 경우가 많다.

또한:

* High Risk Transactions
* drill-down investigation
* reviewer queue

구조와도 가장 자연스럽게 연결된다.

## 현재 단계에서 허용할 범위

```yaml

modulus

```

---

## Config화하지 않을 대상

* 자유 Python expression
* SQL parser
* generic DSL
* 동적 코드 실행 구조

---

## 설계 원칙

현재 단계에서는:

```text
Parameterized Rule Structure
```

까지 구현한다.

```text

```

은 구현하지 않는다.
