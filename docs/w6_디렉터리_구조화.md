# w6 — 프로젝트 디렉터리 구조화

## 배경

Week 1~5까지 기능 구현 중심으로 작업이 진행되면서 폴더 구조가 다음과 같이 산재해 있었다.

- `병합과정/`, `정제과정/`, `탐색과정/`, `excel_output/` — 노트북과 데이터가 같은 폴더에 혼재
- `data/` — 정제 완료 데이터만 존재, 중간 산출물은 각 과정 폴더에 분산
- `docs/` — 모든 MD 파일이 `week0X_` 접두사로 flat하게 나열
- `dataset_git/` — 버전 스냅샷 파일이 `01_/02_/03_` 접두사로 루트에 나열

이 구조는 단기 작업 중에는 문제가 없었지만, 아카이브·공유·GitHub 업로드 시점에서 다음 문제를 드러냈다.

1. **노트북과 데이터의 분리 불가** — 특정 과정 폴더만 공유하면 다른 단계의 입출력 파일을 찾을 수 없음
2. **데이터 단계 불명확** — 어떤 파일이 어느 파이프라인 단계의 산출물인지 폴더 이름만으로 불분명
3. **docs 탐색 비효율** — 주차가 늘어날수록 flat 목록이 길어지고, 파일 간 연관을 파악하기 어려움

## 핵심 설계 원칙

### 1. 노트북과 데이터의 완전 분리

노트북(`.ipynb`)은 **코드**이고, CSV/Excel은 **데이터**다. 이 둘을 같은 폴더에 두면 둘 중 하나만 공유하거나 버전 관리하기 어려워진다. `notebooks/`는 코드 레이어, `data/`는 데이터 레이어로 역할을 명확히 분리했다.

### 2. 파이프라인 단계를 폴더 이름에 명시 (`wN_`)

데이터 파일 하나가 어느 단계의 산출물인지 폴더 이름만 보고 알 수 있도록 `w2_raw → w3_interim → w4_processed → w5_output` 순서를 부여했다. 노트북·문서도 동일한 `wN_` 접두사를 사용해 **코드·데이터·문서의 주차 대응 관계**를 즉시 파악 가능하게 했다.

### 3. `dataset_git/`을 `data/`의 거울 구조로

`data/` 안에서 GitHub에 올릴 수 있는 파일(대용량 원본 SAP 파일이나 비공개 Excel 제외)만 추린 것이 `dataset_git/`다. `data/`와 동일한 하위 폴더 구조(`w2_raw/`, `w3_interim/`, `w4_processed/`, `w5_output/`)를 사용하므로, 로컬 경로와 저장소 경로의 대응이 명확하다.

### 4. 설정 파일은 `config/`로 독립

`config_clean.yaml`은 코드(`notebooks/`)도 데이터(`data/`)도 아니다. 클라이언트가 달라질 때 이 파일만 교체하면 파이프라인 전체가 대응되는 구조이므로, `config/`를 별도 레이어로 두어 교체 지점을 명확히 했다.

## 경로 참조 업데이트 내역

구조 변경에 따라 다음 파일의 경로 상수를 수정했다.

| 파일 | 수정 내용 |
|------|-----------|
| `app.py` | `"data/master_gl_clean.csv"` → `"data/w4_processed/master_gl_clean.csv"` (동일하게 account_master) |
| `notebooks/w2_merge.ipynb` | `BASE_DIR = Path().resolve()` → `INPUT_DIR = PROJECT_ROOT/"data"/"w2_raw"`, `OUTPUT_DIR = PROJECT_ROOT/"data"/"w3_interim"` |
| `notebooks/w3_clean.ipynb` | `CONFIG_FILE = 'config_clean.yaml'` → `'../config/config_clean.yaml'` |
| `notebooks/w4_analyzer.ipynb` | `os.path.join(PROJECT_ROOT, "data", "*.csv")` → `"data", "w4_processed", "*.csv"` |
| `notebooks/w5_excel_export.ipynb` | `Path("../data/*.csv")` → `Path("../data/w4_processed/*.csv")` |
