## 3. 계정 마스터 생성 (an.ipynb)

### 3-1. 목적

`master_gl_clean.csv`에는 개별 계정코드(`acc_code`)와 계정명(`acc_name`)만 존재하며, 대분류·중분류 계층 정보가 없다. Streamlit UI에서 계층별 탐색(대분류 → 중분류 → 계정)을 구현하려면 이 계층 정보를 담은 별도 마스터 파일이 필요하다.

### 3-2. Prefix 기반 계층 분류 규칙

SAP 계정코드는 숫자 앞자리가 계정의 성격을 나타내는 구조다. 이를 활용하여 자동으로 계층을 생성한다.

```python
acc_code_clean = acc_code.lstrip('0')   # 앞자리 0 제거 (정렬·비교용)
lv1_code = acc_code_clean[0]            # 첫 번째 자리 → 대분류
lv2_code = acc_code_clean[:3]           # 처음 3자리   → 중분류
```

생성 직후에는 `lv1_name = "(임시)1"`, `lv2_name = "(임시)113"` 형태의 임시값이 채워지며, 사용자가 CSV를 열어 수동으로 한글 명칭을 입력한다.

**실제 편집 결과 예시**

| lv1_code | lv1_name | lv2_code | lv2_name |
|----------|---------|----------|---------|
| 1 | 자산 | 113 | 현금 및 예금 |
| 1 | 자산 | 121 | 재공품 |
| 2 | 부채 및 조정계정 | 231 | 가격차이 손실 |
| 5 | 매출 | 500 | 매출액 |
| 6 | 판매비와관리비 | 600 | 인건비 |

### 3-3. 검증 항목

| 검증 | 기준 | 판정 |
|------|------|------|
| 마스터 행 수 = unique acc_code 수 | 63개 | ✅ 일치 |
| load_data() 조인 후 행 수 = GL 원본 행 수 | 332,103행 | ✅ 일치 |
| lv1_code 누락 건수 | 0건 | ✅ 전 행 조인됨 |

## 4. analyzer.py — 탐색 함수 모듈

UI 코드(`app.py`)와 데이터 처리 로직을 완전히 분리하여, 함수 단독으로 테스트 가능하도록 설계했다.

### 함수 목록

| 함수 | 역할 |
|------|------|
| `generate_master(df, output_path)` | GL 데이터에서 계정 마스터 생성 및 CSV 저장 |
| `load_data(gl_path, master_path)` | GL + 마스터 left join, post_date datetime 변환 |
| `get_account_list(master, level, parent)` | 계층별 계정 목록 반환 |
| `get_ledger(df, acc_code, level, level_code, date_range)` | 계정/그룹 원장 조회 |
| `get_journal_entry(df, doc_no)` | 전표 분개 조회 (차변/대변 분리) |
| `get_related_accounts(df, doc_no)` | 특정 전표의 연관 계정 목록 |
| `calculate_balance(df, acc_code, level, level_code)` | 차변 합계 / 대변 합계 / 순잔액 계산 |

### 설계 원칙

- **UI 독립성**: 모든 함수는 DataFrame을 입력받고 DataFrame 또는 dict를 반환 → Jupyter·Streamlit·pytest 어디서든 호출 가능
- **레벨 통합 처리**: `get_ledger`와 `calculate_balance`는 `acc_code` 단건 또는 `level + level_code` 그룹 롤업을 동일 인터페이스로 처리
- **None 안전 처리**: `date_range`, `parent` 등 Optional 인수는 None 시 전체 조회로 폴백

### 5-2. 탐색 흐름 (드릴다운)

```
사이드바에서 계정 선택 → [조회] 클릭
  → 원장 테이블 표시 (행별 전표 목록)
    → 행 클릭 → 전표 분개 상세 표시 (차변/대변 라인 버튼)
      → 분개 내 계정 버튼 클릭 → 해당 계정 원장으로 이동
        → 히스토리에 이전 계정 기록 → [← 이전] 버튼으로 복귀 가능
```

### 5-3. 주요 구현 포인트

**session_state 기반 상태 관리**  
Streamlit은 매 상호작용마다 전체 스크립트를 재실행하므로, `st.session_state`에 현재 조회 계정·선택 전표·탐색 히스토리를 보존한다.

```python
st.session_state.target_level  # 1 / 2 / 3
st.session_state.target_code   # 현재 조회 중인 계정/그룹 코드
st.session_state.selected_doc  # 선택된 전표번호
st.session_state.history       # 탐색 경로 스택 (최대 10개)
```

**행 선택 이벤트 처리**  
`st.dataframe(on_select="rerun", selection_mode="single-row")`로 테이블 행 클릭 시 자동으로 스크립트 재실행, 선택 인덱스를 `session_state`에서 읽어 전표 조회 실행.

**계층별 조회 통합**  
`get_ledger()`와 `calculate_balance()`는 레벨에 따라 분기:
- level 3(세분류): `acc_code` 단건 필터
- level 1/2(대·중분류): `lv1_code` 또는 `lv2_code` 기준 그룹 롤업

**캐싱**  
`@st.cache_data`로 GL + 마스터 조인 결과를 메모리에 캐싱하여, 매 조작 시 CSV 재로드를 방지.

## 6. 시행착오 및 배운 점

### 6-1. Streamlit 실행 모델의 이해

일반적인 프로그램은 이벤트에 반응하여 해당 부분만 업데이트된다. Streamlit은 다르다. 버튼 클릭, 선택박스 변경 등 **모든 상호작용마다 전체 Python 파일을 처음부터 재실행**한다.

따라서 "이 버튼을 눌렀을 때 이 변수를 바꿔라"는 방식으로는 상태를 유지할 수 없고, 모든 상태는 `st.session_state`에 저장해야 한다. 이 구조를 이해하는 것이 Streamlit 개발의 핵심이다.

### 6-2. get_journal_entry() 내부 로직

처음 코드를 작성할 때 `_rows()` 함수 안에서 `is_debit` 변수를 `mask.name`으로 판별하려는 잘못된 구조가 있었다. 결국 차변/대변 마스크를 별도로 분리하고, 각각에 대해 독립적인 리스트 컴프리헨션으로 재작성했다.

### 6-3. 계층 탐색과 개별 계정 탐색의 인터페이스 통합

`get_ledger()`를 처음 설계할 때 레벨 1/2 그룹 조회와 레벨 3 단건 조회를 별도 함수로 나눌지 고민했다. 결국 Optional 인수(`acc_code` 또는 `level + level_code`)로 하나의 함수에서 분기 처리하는 방식을 선택했고, `app.py`에서 호출 코드가 훨씬 단순해졌다.

### 6-4. 전체 학습 소감

이번 주 내용은 예상보다 난이도가 높았다. Streamlit의 실행 모델, session_state 패턴, 함수형 설계, 타입 힌팅 등 여러 개념이 동시에 등장했고, 코드의 의미를 완전히 파악하기보다 **구조를 이해하며 필사하는 방식**으로 진행했다. 그 과정에서 각 함수가 어떤 역할을 담당하는지, UI 레이어와 데이터 레이어가 어떻게 분리되는지는 파악할 수 있었다.

계정 마스터 파일을 만드는 과정에서 해당 데이터셋 전처리의 부실함을 확인했다. 우선 BS/PL 계정 분류가 적절치 못하다. 계정 마스터 파일을 만드는 것도 정교하지 못하다. 코드 내에서 처리를 하는 과정에서 존재하지 않은 '113000' 계정번호의 land가 생성된 것이 그 예시다. 따라서, 관련 계정 마스터를 클라이언트에게 요청하거나 엑셀을 활용하여 직접 정교하게 만드는 것이 최종 결과물의 품질을 끌어올리는 방법일 수 있겠다고 느꼈다.

Streamlit이 인터랙티브한 탐색 도구를 로컬 환경에서 빠르게 구현하는 데 매우 효과적인 도구임을 체감했다. 실무에서 감사 데이터를 탐색할 때 이 구조를 그대로 활용할 수 있을 것으로 보인다.
